# Specs

Committed design documents that back implementation work.

A spec lays out the problem, goals, non-goals, and design for a non-trivial change before code is written. It's the contract that an implementation plan executes against and the artifact a future maintainer can read to understand *why* the code looks the way it does.

Files here are durable: kept under version control, referenced from PRs and code comments, and updated only when the design itself materially changes.

Distinct from:
- [`../ideas/`](../ideas/) — design proposals we've considered but deferred; preserved for context, not committed to.
- [`../plans/`](../plans/) — gitignored, ephemeral implementation plans that translate a spec into bite-sized executable tasks.
