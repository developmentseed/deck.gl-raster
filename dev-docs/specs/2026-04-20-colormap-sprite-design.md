# Colormap Sprite Design

## Problem

The `Colormap` shader module currently takes a single `sampler2D` bound to a
1D LUT — one colormap per layer, one texture per colormap. A separate script
(`packages/deck.gl-raster/scripts/generate_colormaps.py`) produces a per-colormap
PNG file for the matplotlib / rio-tiler colormap catalog, so a consumer who
wants to offer "pick any matplotlib colormap" has to ship ~50 ~120-byte PNGs
and issue a new network fetch (and a texture upload) every time the user
switches colormap in the UI.

We want to bundle the full catalog into a single GPU asset so that:

- There is one fetch and one texture upload for the whole catalog.
- Switching the active colormap at runtime is a uniform update, not a texture
  rebind or re-upload — the fact that "everything is happening on the GPU"
  shows up in how the API feels.
- Multiple layers rendered in the same frame can each pick a different
  colormap from the same shared texture without duplicating VRAM.

## Goals

- Replace the single-colormap `sampler2D` with a single unified module backed
  by a `sampler2DArray`, where each layer of the array is one colormap.
- Ship a generated `colormaps.png` sprite (N rows × 256 pixels, RGBA8)
  covering the rio-tiler / matplotlib catalog, together with a generated
  TypeScript `COLORMAP_INDEX` constant and `ColormapName` union type.
- Provide a runtime helper that loads the sprite (from a URL or already-decoded
  bytes) into a `Texture2DArray`.
- Keep the "I only have one colormap" case simple: `colormapIndex` defaults
  to `0`, and a 1-layer sprite is a valid input.
- Preserve the existing `reversed` uniform so flipping (`viridis_r`) stays a
  uniform update.

## Non-Goals

- **Shipping the sprite inlined in the JS bundle.** The PNG lives in `dist/`
  and consumers wire it up via their bundler of choice (URL import, arraybuffer
  import, manual fetch). We do not want to bloat the library's JS bundle with
  ~50KB of binary colormap data.
- **User-authored colormaps at runtime.** The API accepts any
  `Texture2DArray`, so a user can build their own — but generating sprites on
  the fly from GLSL-less data is out of scope for this module.
- **Bilinear interpolation between colormap layers.** Switching colormaps is
  discrete; there is no "blend from viridis to plasma" feature.
- **Changing the PNG regeneration pipeline.** `generate_colormaps.py` stays
  a manual Python script, run on demand; the generated artifacts are checked
  in.
- **A backwards-compatible `sampler2D` fallback.** The module is pre-1.0; we
  take the breaking change and migrate in the CHANGELOG.

## Design

### Shader module shape

`packages/deck.gl-raster/src/gpu-modules/colormap.ts` — same file, new shape.

```ts
import type { Texture } from "@luma.gl/core";
import type { ShaderModule } from "@luma.gl/shadertools";

/** Props for the {@link Colormap} shader module. */
export type ColormapProps = {
  /**
   * The colormap sprite as a 2D array texture. Each layer of the array is
   * one 256×1 RGBA8 colormap. Build from the shipped `colormaps.png` with
   * {@link loadColormapSprite}, or bring your own.
   */
  colormapTexture: Texture;
  /**
   * Which layer of `colormapTexture` to sample. Pass values from the
   * generated {@link COLORMAP_INDEX} for the shipped sprite, or any valid
   * layer index for a custom sprite. Defaults to `0`.
   */
  colormapIndex?: number;
  /**
   * When true, samples the colormap in reverse — equivalent to matplotlib's
   * `_r` suffix (e.g. `viridis_r`). Defaults to false.
   */
  reversed?: boolean;
};

const MODULE_NAME = "colormap";

export const Colormap = {
  name: MODULE_NAME,
  fs: `\
uniform ${MODULE_NAME}Uniforms {
  int colormapIndex;
  float reversed;
} ${MODULE_NAME};
`,
  inject: {
    "fs:#decl": `uniform sampler2DArray colormapTexture;`,
    "fs:DECKGL_FILTER_COLOR": /* glsl */ `
      float idx = mix(color.r, 1.0 - color.r, ${MODULE_NAME}.reversed);
      color = texture(
        colormapTexture,
        vec3(idx, 0.5, float(${MODULE_NAME}.colormapIndex))
      );
    `,
  },
  uniformTypes: {
    colormapIndex: "i32",
    reversed: "f32",
  },
  getUniforms: (props: Partial<ColormapProps>) => {
    return {
      colormapTexture: props.colormapTexture,
      colormapIndex: props.colormapIndex ?? 0,
      reversed: props.reversed ?? false,
    };
  },
} as const satisfies ShaderModule<ColormapProps>;
```

The `v` coordinate is `0.5` rather than `0.0` because each layer is a single
texel tall; `0.5` centers the sample in that texel and is robust to any
filtering mode.

### Asset layout

Generated by the Python script under `packages/deck.gl-raster/src/gpu-modules/`:

- `colormaps.png` — `N × 256 × 4` RGBA8 image. Row `i` is colormap `i` in the
  same order as `COLORMAP_INDEX`.
- `colormap-names.ts` — generated TypeScript:
  ```ts
  // Generated by scripts/generate_colormaps.py. Do not edit by hand.
  export const COLORMAP_INDEX = {
    accent: 0,
    afmhot: 1,
    // ...
    viridis: 48,
    // ...
  } as const;

  export type ColormapName = keyof typeof COLORMAP_INDEX;
  ```

`tsc --build` copies `colormap-names.ts` into `dist/` as part of normal
compilation. The PNG is copied by a small copy step in the package's
`build` script (e.g. a Node one-liner invoked after `tsc`) so it lands at
`dist/gpu-modules/colormaps.png`.

### Package exports

Expose the PNG as a subpath so consumers can use bundler-native imports:

```json
"./gpu-modules/colormaps.png": "./dist/gpu-modules/colormaps.png"
```

Example consumer patterns (documented in the module's JSDoc):

```ts
// Vite / webpack with url-loader: ?url returns a string URL
import colormapsUrl from "@developmentseed/deck.gl-raster/gpu-modules/colormaps.png?url";
const { texture, index } = await loadColormapSprite(device, colormapsUrl);
```

```ts
// Bundler configured to treat .png as arraybuffer: pass the buffer directly
import colormapsBuffer from "@developmentseed/deck.gl-raster/gpu-modules/colormaps.png";
const { texture, index } = await loadColormapSprite(device, colormapsBuffer);
```

### Loader helper

New file: `packages/deck.gl-raster/src/gpu-modules/load-colormap-sprite.ts`.

```ts
import type { Device, Texture } from "@luma.gl/core";

/**
 * Load a colormap sprite into a 2D array texture on the given device.
 *
 * Accepts either:
 * - a URL string (the helper issues a `fetch` and decodes the response), or
 * - an already-decoded source (`ArrayBuffer`, `Uint8Array`, `ImageBitmap`,
 *   or `ImageData`) — useful when the consumer's bundler has already handled
 *   the binary asset.
 *
 * The returned texture has `dimension: "2d-array"` with one layer per row of
 * the source image. Use it as the `colormapTexture` prop of the `Colormap`
 * shader module.
 *
 * Name → layer index mappings are not this helper's concern: for the shipped
 * sprite, import {@link COLORMAP_INDEX} directly; for custom sprites, the
 * consumer owns whatever mapping they need.
 */
export async function loadColormapSprite(
  device: Device,
  source: string | ArrayBuffer | Uint8Array | ImageBitmap | ImageData,
): Promise<Texture>;
```

Internal implementation responsibilities:

1. Normalize `source` to `ImageData` (fetch + `createImageBitmap` + canvas if
   it's a URL or a raw buffer).
2. Compute `layerCount = imageData.height`; throw if `imageData.width` is not
   256.
3. Slice the image into `layerCount` `Uint8Array` rows and upload as a
   `Texture` with `dimension: "2d-array"`, `format: "rgba8unorm"`, size
   `[256, 1, layerCount]`, and `mipmaps: false`. Filters: `mag: "linear"`,
   `min: "linear"` — safe because there is only one texel in the `v`
   direction and we always sample at `v = 0.5`.
4. Return the `Texture`.

Keeping name→index separate from the loader avoids a footgun where a
consumer loads a custom sprite and silently inherits the default catalog's
mapping. Importing `COLORMAP_INDEX` directly is equally ergonomic for the
common case:

```ts
import { Colormap, COLORMAP_INDEX, loadColormapSprite }
  from "@developmentseed/deck.gl-raster/gpu-modules";

const texture = await loadColormapSprite(device, colormapsUrl);
// ...
{ module: Colormap, props: {
    colormapTexture: texture,
    colormapIndex: COLORMAP_INDEX.viridis,
  } }
```

### Generator script

Rewrite `packages/deck.gl-raster/scripts/generate_colormaps.py` to emit a
single PNG and a single TS file instead of N separate PNGs:

```python
# For each name in rio_tiler.cmap_data.cmap_list (sorted):
#   row = make_lut(get_colormap(name))          # shape (256, 4) uint8
#   rows.append(row)
# sprite = np.stack(rows, axis=0)               # shape (N, 256, 4) uint8
# imageio.imwrite(out_dir / "colormaps.png", sprite)
#
# with open(out_dir / "colormap-names.ts", "w") as f:
#     f.write("// Generated by scripts/generate_colormaps.py...\n")
#     f.write("export const COLORMAP_INDEX = {\n")
#     for i, name in enumerate(sorted_names):
#         f.write(f"  {name}: {i},\n")
#     f.write("} as const;\n\n")
#     f.write("export type ColormapName = keyof typeof COLORMAP_INDEX;\n")
```

Output directory defaults to `packages/deck.gl-raster/src/gpu-modules/`. The
script is run manually and its outputs are committed, so the npm `build`
flow does not depend on Python.

The generated artifacts are **not** excluded from git (`.gitignore`) — they
are reviewed like any other source file.

## Architecture Overview

```
Python script (manual, on demand)
  │
  ├─ colormaps.png              ← N × 256 × 4 RGBA8 sprite
  └─ colormap-names.ts          ← { viridis: 48, ... } + ColormapName type

                                    committed to src/gpu-modules/

tsc build
  │
  ├─ copies colormap-names.ts → dist/gpu-modules/colormap-names.js + .d.ts
  └─ copy step:   colormaps.png → dist/gpu-modules/colormaps.png

Consumer at runtime
  │
  ├─ import colormapsUrl from "…/gpu-modules/colormaps.png?url"
  ├─ const { texture, index } = await loadColormapSprite(device, colormapsUrl)
  │
  └─ per-layer render pipeline:
        { module: Colormap, props: {
            colormapTexture: texture,
            colormapIndex: index.viridis,   // or COLORMAP_INDEX.viridis
            reversed: false,
          } }

Shader (per fragment)
  │
  color = texture(colormapTexture,
                  vec3(idx, 0.5, float(colormap.colormapIndex)))
```

## Testing

### Unit tests

Existing file `packages/deck.gl-raster/tests/gpu-modules/colormap.test.ts` is
updated; a new `load-colormap-sprite.test.ts` covers the loader. No GPU
needed — mock `Device` / `Texture` with `as any` as elsewhere in this repo.

`Colormap` module:

- `getUniforms` returns the provided `colormapTexture`, `colormapIndex`, and
  `reversed` unchanged when all are supplied.
- `getUniforms` defaults `colormapIndex` to `0` and `reversed` to `false`
  when omitted.
- Module metadata: `name === "colormap"`, `uniformTypes.colormapIndex ===
  "i32"`, `uniformTypes.reversed === "f32"`.
- Inject hooks: `fs:#decl` declares `uniform sampler2DArray colormapTexture`;
  `fs:DECKGL_FILTER_COLOR` samples with a `vec3(..., 0.5, float(...))`
  coordinate.

`loadColormapSprite`:

- Given a mocked `ImageData` of width 256 and height 3, calls
  `device.createTexture` once with `dimension: "2d-array"`, `format:
  "rgba8unorm"`, width 256, height 1, depth 3.
- Throws when the source's width is not 256.
- Returns the same `Texture` instance that `device.createTexture` produced.

The URL / `ArrayBuffer` / `Uint8Array` branches of the normalization logic
depend on browser-only APIs (`fetch`, `createImageBitmap`, canvas
`getImageData`) that jsdom does not fully implement — they are exercised
via the manual visual verification app, not unit tests.

### Manual visual verification

A small example app (not part of this spec's implementation) loads the
shipped sprite, renders a raster with a `Colormap` shader module, and wires
a `<select>` bound to a React state variable that drives
`props.colormapIndex` on every render. Success criteria:

- Changing the dropdown updates the colormap with no visible fetch delay.
- No new textures are allocated on switch (verify via devtools / luma debug
  counters).
- Two deck.gl layers, each with a different `colormapIndex`, render
  simultaneously with different colormaps.
